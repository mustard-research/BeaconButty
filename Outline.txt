⏺ Beacon Detection on Corporate Networks

  This is a well-established area of defensive security (Network Security Monitoring / NSM). It's highly
   feasible and actively done — let me walk through the technical landscape.

  ---
  How Beacons Behave (What You're Detecting)

  Beacons have characteristic statistical signatures:

  - Periodicity — connections at regular intervals (every N seconds/minutes)
  - Jitter — sophisticated malware adds ±randomness to avoid naive detection
  - Small, consistent payload sizes — just a check-in, not bulk data transfer
  - Long-lived relationship — same source IP → same destination, repeated over hours/days
  - Low data volume — distinguishes from legitimate streaming traffic

  ---
  Network Position for a Detector

  [Internal Hosts] → [Switch/TAP] → [Router/NAT] → [Internet]
                           ↑
                     [Beacon Detector]

  Best capture points:

  ┌──────────────────────────────┬──────────────────────────┬───────────────────────────────────────┐
  │           Position           │           Pros           │                 Cons                  │
  ├──────────────────────────────┼──────────────────────────┼───────────────────────────────────────┤
  │ Inline between switch and    │ Sees all traffic before  │ Single point of failure               │
  │ router                       │ NAT                      │                                       │
  ├──────────────────────────────┼──────────────────────────┼───────────────────────────────────────┤
  │ SPAN/mirror port on switch   │ Passive, no risk         │ May miss traffic depending on switch  │
  │                              │                          │ config                                │
  ├──────────────────────────────┼──────────────────────────┼───────────────────────────────────────┤
  │ On the router itself         │ Sees everything          │ Needs router access, CPU overhead     │
  ├──────────────────────────────┼──────────────────────────┼───────────────────────────────────────┤
  │ Per-VLAN on managed switch   │ Granular, segmented      │ More complex setup                    │
  └──────────────────────────────┴──────────────────────────┴───────────────────────────────────────┘

  For WiFi: the same SPAN/TAP approach works if you capture at the AP controller or the uplink switch
  rather than over the air.

  ---
  Core Detection Techniques

  1. Periodicity Analysis (Most Effective)

  The fundamental signal. For each (src_ip, dst_ip, dst_port) tuple, collect inter-arrival times and
  look for regularity.

  import numpy as np
  from scipy import signal

  def beacon_score(connection_times):
      """
      Returns a score 0-1 indicating likelihood of beaconing.
      Higher = more beacon-like.
      """
      if len(connection_times) < 6:
          return 0.0

      # Inter-arrival times
      intervals = np.diff(sorted(connection_times))

      # Low coefficient of variation = regular intervals
      cv = np.std(intervals) / np.mean(intervals)
      regularity_score = max(0, 1 - cv)

      # FFT to detect dominant frequency
      fft = np.abs(np.fft.rfft(intervals))
      dominant_power = fft[1:].max() / (fft[1:].mean() + 1e-9)
      frequency_score = min(1.0, dominant_power / 10.0)

      return 0.6 * regularity_score + 0.4 * frequency_score

  2. Payload Size Consistency

  Beacons tend to send similar-sized packets each time. Compute variance of bytes_sent and
  bytes_received per flow group over time.

  3. Destination Reputation / Novelty

  - New domains not seen before on the network
  - Low Alexa/Tranco rank
  - Recently registered domains (check via passive DNS or WHOIS)
  - IP geolocation anomalies
  - Self-signed TLS certificates, unusual cipher suites

  4. JA3/JA3S TLS Fingerprinting

  TLS clients have fingerprints based on their handshake parameters. Many malware families have known
  JA3 hashes. You can match against threat intel feeds.

  5. DNS Beaconing Detection

  Some C2 uses DNS (queries encode data, responses encode commands):

  - High query rate to a single domain
  - Long/random-looking subdomains
  - Unusual query types (TXT, NULL records)
  - Low TTL responses
  - Domain Generation Algorithm (DGA) detection via entropy analysis

  import math

  def domain_entropy(domain):
      """High entropy subdomains suggest DGA or DNS tunneling."""
      label = domain.split('.')[0]
      prob = [label.count(c) / len(label) for c in set(label)]
      return -sum(p * math.log2(p) for p in prob)

  # DGA domains typically have entropy > 3.5

  ---
  Practical Implementation Stack

  Capture layer:
  - libpcap / AF_PACKET sockets on Linux for raw capture
  - Zeek (formerly Bro) — generates structured connection logs automatically
  - Suricata — can do both signature and statistical detection

  Flow aggregation:
  - Parse conn.log from Zeek, or use nfdump/ntopng for NetFlow
  - Group by (src, dst, port, proto) with time-windowed buckets

  Analysis layer:
  - Python with pandas/numpy/scipy for statistical analysis
  - Redis or TimescaleDB for storing per-flow time series
  - Run analysis on sliding windows (e.g., score each flow over last 24h)

  Alerting:
  - Output to a SIEM (Elastic/Splunk/Wazuh)
  - Or directly to a ticketing system

  ---
  Key Challenges

  ┌───────────────────────────────┬─────────────────────────────────────────────────────────────────┐
  │           Challenge           │                           Mitigation                            │
  ├───────────────────────────────┼─────────────────────────────────────────────────────────────────┤
  │ Encrypted traffic (HTTPS/TLS) │ Focus on metadata (timing, size, frequency) not content — still │
  │                               │  works                                                          │
  ├───────────────────────────────┼─────────────────────────────────────────────────────────────────┤
  │ Jittered beacons              │ Use FFT and autocorrelation rather than simple variance         │
  ├───────────────────────────────┼─────────────────────────────────────────────────────────────────┤
  │ Legitimate periodic traffic   │ Build a baseline allowlist (NTP, update checks, telemetry)      │
  ├───────────────────────────────┼─────────────────────────────────────────────────────────────────┤
  │ High traffic volume           │ Aggregate to flow-level data first, don't analyze raw packets   │
  ├───────────────────────────────┼─────────────────────────────────────────────────────────────────┤
  │ HTTPS beacons over CDN IPs    │ Combine with JA3, SNI hostname, and behavioral clustering       │
  ├───────────────────────────────┼─────────────────────────────────────────────────────────────────┤
  │ Low-frequency beacons         │ Requires multi-day retention of flow metadata                   │
  │ (hourly+)                     │                                                                 │
  └───────────────────────────────┴─────────────────────────────────────────────────────────────────┘

  ---
  Existing Open-Source Tools

  - https://github.com/activecm/rita (Real Intelligence Threat Analytics) — specifically designed for
  this, works with Zeek logs
  - https://zeek.org/ — foundational platform, has beacon detection scripts
  - https://github.com/Cyb3rWard0g/HELK — hunting platform built on ELK + Zeek
  - https://github.com/activecm/BeaKer — Zeek + RITA packaged together

  ---
  Summary

  Very feasible. The key insight is that you don't need deep packet inspection — beacon detection works
  entirely on connection metadata (timing, frequency, size distributions), which survives encryption. A
  network tap + Zeek + RITA is a production-quality stack that many companies deploy today. The hardest
  problems are tuning the allowlist to reduce false positives from legitimate periodic software (Windows
   Update, Chrome telemetry, etc.) and handling very-low-frequency beacons that require days of data to
  become statistically significant.

  ---
  Does Zeek work on the outgoing (NAT) pipe of a router?

  Yes — but where you place the tap changes what you can see.

  LAN-side tap (strongly preferred)

  [Internal hosts 192.168.x.x]
           |
     [Managed Switch] ← SPAN port → [Pi eth1]  ← TAP HERE
           |
        [Router/NAT]
           |
        [Internet]

  - Zeek sees the real internal IP (e.g. 192.168.1.45) as the source
  - You know which machine is beaconing — this is what RITA needs
  - This is what the implementation above assumes

  WAN-side tap (after NAT)

  [Managed Switch]
         |
     [Router/NAT]
         |
    [Pi eth1]  ← TAP HERE  (or on modem uplink)
         |
     [Internet]

  - Zeek only sees the router's public IP as the source of every connection
  - You can detect that beaconing is happening, but not which internal host
  - Less useful for incident response

  Recommendation: Tap between the switch and the router (LAN-side). If you only have a consumer router
  with no SPAN port, the options are:

  ┌───────────────────────┬─────────────────────────────────────────────────────────────────────────┐
  │        Option         │                                   How                                   │
  ├───────────────────────┼─────────────────────────────────────────────────────────────────────────┤
  │ Inline Pi bridge      │ Pi sits between switch and router; bridges eth0↔eth1, captures on eth1. │
  │                       │  Single point of failure.                                               │
  ├───────────────────────┼─────────────────────────────────────────────────────────────────────────┤
  │ Managed switch SPAN   │ Mirror all switch ports to the Pi. Best option — passive.               │
  │ port                  │                                                                         │
  ├───────────────────────┼─────────────────────────────────────────────────────────────────────────┤
  │ OpenWrt/pfSense       │ Run packet capture directly on the router's mirror interface.           │
  │ router                │                                                                         │
  ├───────────────────────┼─────────────────────────────────────────────────────────────────────────┤
  │ Network tap hardware  │ Passive copper tap (e.g. Dualcomm ETAP-2003) — completely invisible, no │
  │                       │  failure risk.                                                          │
  └───────────────────────┴─────────────────────────────────────────────────────────────────────────┘

  Inline Pi bridge setup (if you have no managed switch)

  If you need to go inline, add this to your setup:

  # On the Pi: bridge eth0 (to router) and eth1 (to switch), capture on eth1
  apt-get install -y bridge-utils
  brctl addbr br0
  brctl addif br0 eth0 eth1
  ip link set br0 up
  # Zeek watches eth1 in promiscuous mode — sees all traffic crossing the bridge

  Then in node.cfg, set interface=eth1 (the switch-facing side) — Zeek will see the original internal
  IPs before they hit the router's NAT.

⏺ Good question — let me step back from the bridge implementation, because the architecture decision
  here is more important.

  The core problem with your current setup

  [WiFi clients]  ─┐
                    ├─ [Netgear internal bus] ─ [NAT] ─── [ISP]
  [Switch clients] ─┘         ↑
                        inaccessible from outside

  The internal switching and WiFi happen inside the Netgear's chip — you can't tap that bus. The only
  external port is the WAN, which is post-NAT. Zeek on the WAN sees only your router's single public IP
  as the source of every connection. You can detect that beaconing is happening but not which internal
  host is responsible.

  ---
  Three architectures, one clear winner

  Option A — Managed switch with SPAN port (passive, non-breaking)

  [WiFi clients on Netgear WiFi]
             |
      [Netgear in AP mode]  ──┐
                               ├── [Managed switch] ──── [ISP router/modem]
      [Wired clients]  ────────┘         |
                                    SPAN port
                                         │
                                      [Pi eth1]
                                      Zeek captures

  - Netgear is reconfigured to AP-only (disable its DHCP/NAT, use one of its LAN ports as uplink)
  - A cheap managed switch handles all forwarding
  - SPAN port mirrors all traffic to the Pi — completely passive
  - Hardware cost: ~£25-35 for a TP-Link TL-SG108E (8-port, supports port mirroring)
  - Risk: Zero. The Pi is not in the data path.
  - Gap: WiFi traffic between clients (client-to-client on same AP) won't be visible if the Netgear
  doesn't bridge it through the switch — but outgoing internet traffic will be.

  Option B — Pi as the router (best visibility, most control)

  [Wired clients]  ──┐
                      ├── [Dumb switch] ── eth1 (Pi LAN) ← Zeek captures here
  [AP in AP mode] ───┘                        │
                                          Pi (NAT + Zeek)
                                              │
                                            eth0 ── [ISP modem/connection]

  - The Pi replaces the Netgear's routing/NAT function entirely
  - Zeek runs on the Pi's LAN interface — sees every internal host's real IP
  - Netgear (or a cheap replacement AP) used in AP-only mode
  - Pi runs iptables NAT and optionally a DHCP server (or use dnsmasq)
  - Hardware cost: Just a cheap unmanaged switch (~£10) if you already have two Pi NICs
  - Risk: If the Pi crashes, internet goes down (mitigated by watchdog)
  - Visibility: Complete — every connection from every host

  Option C — Inline Pi bridge on the WAN (what we were building)

  [Netgear (NAT + WiFi + switch)]
              |
            [Pi bridge]  ← Zeek captures here (POST-NAT)
              |
            [ISP]

  - All traffic appears as the router's single public IP
  - Can still detect beaconing by timing/frequency, but can't identify which internal host
  - Only useful for "is anything on this network beaconing?" not "which machine?"
  - Verdict: least useful for your goals

  ---
  Recommendation: Option A first, upgrade to B later

  Start with Option A — it's non-breaking and cheap:

  scripts/wan-watchdog.sh

  Runs every 5 minutes. Checks WAN interface has an IP, then pings 1.1.1.1 and 8.8.8.8. After 3
  consecutive failures (~15 min) it attempts a DHCP renewal. Logs to /var/log/beaconbutty/watchdog.log
  and syslog. Does not revert — in router mode there's no fallback topology.

  iptables firewall posture

  INPUT:    DROP by default
            ACCEPT lo, ESTABLISHED, all from LAN, ICMP echo
  FORWARD:  LAN → WAN (new + established)
            WAN → LAN (established only — no unsolicited inbound)
  NAT:      MASQUERADE on WAN

❯ how does the range through walls on the  TP-Link EAP773  compare with the other options for the       
  ap                                                                                                 
                                                                                                        
⏺ Honest answer: the EAP773 is a solid AP but it has a meaningful weakness for through-wall range
  compared to some of the alternatives, for a specific reason.                                          
                                                                                                        
  The issue: 2×2 MIMO on 5 GHz
                                                                                                        
  The EAP773 uses a 2×2 antenna configuration on each band. The competitors at similar or slightly
  higher price points use 4×4 on 5 GHz, which matters a lot for wall penetration because more streams
  enable better beamforming — the AP can concentrate signal toward a client rather than radiating it
  equally in all directions.

  ┌──────────────────────────┬─────────┬───────┬───────┬───────┐
  │            AP            │ 2.4 GHz │ 5 GHz │ 6 GHz │ Price │
  ├──────────────────────────┼─────────┼───────┼───────┼───────┤
  │ TP-Link EAP773           │ 2×2     │ 2×2   │ 2×2   │ ~£130 │
  ├──────────────────────────┼─────────┼───────┼───────┼───────┤
  │ TP-Link EAP783           │ 4×4     │ 4×4   │ 4×4   │ ~£200 │
  ├──────────────────────────┼─────────┼───────┼───────┼───────┤
  │ Ubiquiti U7 Pro          │ 4×4     │ 4×4   │ 2×2   │ ~£200 │
  ├──────────────────────────┼─────────┼───────┼───────┼───────┤
  │ Ubiquiti U6 Pro (WiFi 6) │ 4×4     │ 4×4   │ —     │ ~£120 │
  └──────────────────────────┴─────────┴───────┴───────┴───────┘

  The 6 GHz problem (affects all WiFi 7 APs equally)

  The new 6 GHz band that makes WiFi 7 fast is also its Achilles heel through walls. The physics is
  straightforward: higher frequency = shorter wavelength = more absorption by building materials.
  Concrete, brick, and even plasterboard with foil-backed insulation will kill 6 GHz signal after one
  wall. This affects every WiFi 7 AP equally — it's not a flaw specific to the EAP773.

  In practice, 6 GHz is essentially a short-range, high-speed band for devices in the same room.

  Real-world ranking for through-wall coverage (single AP)

  1. Ubiquiti U6 Pro — consistently rated best-in-class for range from a single AP; 4×4 on 5 GHz with
  mature beamforming firmware; no 6 GHz band means all antenna resources go to 2.4/5 GHz
  2. Ubiquiti U7 Pro — marginally better than U6 Pro on connected throughput, similar real-world range;
  4×4 on 5 GHz
  3. TP-Link EAP783 — 4×4 across all three bands; comparable to the Ubiquiti units but TP-Link's
  beamforming implementation is slightly behind Ubiquiti's in independent reviews
  4. TP-Link EAP773 — noticeably behind the above three on a multi-room/multi-wall deployment due to 2×2
   MIMO

  Practical guidance

  - Small flat or open-plan office (2–4 rooms, few walls): EAP773 is fine and the £70 saving over a U7
  Pro is real money
  - House with multiple floors or thick walls: the U7 Pro or EAP783 will give meaningfully better
  coverage from a single unit
  - Large or awkward space: no single AP will be ideal — the right answer is two cheaper APs (e.g. two
  EAP773s) with Omada roaming configured, which Zeek and RITA handle transparently since both APs are on
   the same subnet

  For the BeaconButty use case specifically, AP choice doesn't affect the security monitoring at all —
  Zeek sees traffic at the router (Pi eth1) regardless of which AP the client connected through.
